# 멀티에이전트 토론 요약 (10라운드)

## 참여자
- 김도현 (백엔드 엔지니어)
- 박서연 (법률/컴플라이언스 전문가)
- 이준호 (프로덕트 매니저)
- 최민지 (보안/안티어뷰즈 엔지니어)
- 정하은 (사업/수익화 분석가)
- 기획자 (서비스 기획자, 직접 참여)

## 라운드별 핵심 논점

### R1: 초기 리스크 식별
- DC 크롤링 안정성 및 법적 리스크
- LLM 캡셔닝 비용 스케일링
- PMF 부재 가능성
- 수익화 어려움

### R2: 기획자 개입 - "본 적 있는 짤을 다시 찾는" 유스케이스
- 서비스 본질: "재발견" → 기존 검색 대비 명확한 우위
- 유저 업로드 방식으로 전환 제안 (법적 리스크 감소)
- 밈 맥락 캡셔닝의 기술적 어려움

### R3: 기획자 개입 - 콜드 스타트 문제
- 유저 업로드는 이용자 없이 불가능
- 브라우저 확장 프로그램으로 반자동 수집 제안
- 특정 카테고리 집중 전략 (500개로 MVP)

### R4: 기획자 개입 - 멀티 소스 + URL 깨짐
- DC 한정 탈피 → X, Pixiv, Gelbooru 등 멀티 소스
- 이미지 URL 깨짐 해결: 썸네일(200x200) 자체 저장 + 원본 링크
- Cloudflare R2 (이그레스 무료) 활용
- 법적으로 "검색 엔진" 포지셔닝 유지

### R5: 기획자 개입 - 창작자 홍보 확장
- AI 이미지 제작 추세 vs 원본 창작자 보호
- 크리에이터 프로필 시스템 → 수익 모델 변화
- "짤과 원작자를 연결하는 검색 엔진" 아이덴티티 확립
- 법적 포지셔닝 강화 (저작권 보호 기여)

### R6: 기획자 개입 - 짤 숏폼 피드
- 숏폼 스와이프 UI로 리텐션 확보
- 벡터 DB 인프라를 추천에 그대로 재활용
- 인터스티셜 광고 모델 → DAU 3,000-5,000이면 서버비 커버
- 검색(유입) + 피드(리텐션) 이중 구조

### R7: 경쟁 분석
- 2짤, 오늘의짤방: 수동 태그, 피드 없음
- 9GAG: 피드 강함, 검색 약함, MAU 1.5억
- SauceNAO: 역이미지 검색, 창작자 출처
- 핵심 발견: 검색 + 피드 + 창작자 연결 결합 서비스는 없음
- CLIP 하이브리드 벡터 방식 제안 (캡셔닝 비용 절감)

### R8: 성공 가능성 판단
- 전원 Conditional Go
- 사이드 프로젝트로 시작, PMF 확인 전 과투자 금지
- 성공 확률: 사이드 프로젝트 15-20%, 풀타임 사업 5% 미만
- 포트폴리오 겸용 권장

### R9: 기획자 개입 - Gelbooru NSFW 리스크
- Gelbooru 제외 (NSFW 비율 과다, 광고 네트워크 정지 리스크)
- Safebooru (SFW 전용) 대체
- 소스 우선순위 재정리: X > 브라우저 확장 > Safebooru > Danbooru > 유저 업로드

### R10: NSFW 수요와 법적 리스크
- NSFW 수요가 압도적으로 높은 것은 사실
- 한국법: "음란물 유포"는 불법이지만, 성인인증 + 접근차단 시 합법
- 전략: SFW로 시작 → 브랜드 확립 후 NSFW 분리 확장
- 리디북스/봄툰 모델 참조

## 최종 합의 사항
1. **사이드 프로젝트로 시작** (풀타임 올인 금지)
2. **Fake door 테스트로 수요 검증 먼저**
3. **CLIP 하이브리드 벡터** (캡셔닝 비용 최소화)
4. **멀티 소스 API** (DC 크롤링 포기)
5. **썸네일 + 원본 링크** (저작권 안전)
6. **검색(유입) + 피드(리텐션)** 이중 구조
7. **창작자 연결로 차별화** 및 법적 보호
8. **SFW 우선, NSFW는 장기 옵션**
